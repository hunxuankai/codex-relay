# 代码质量检查

对最近编写的代码进行综合质量验证，结合规范符合性、跨层安全和提交前检查。

---

## 步骤 1：识别改动

```bash
git diff --name-only HEAD
git status
```

## 步骤 2：读取任务材料和适用规范

按顺序读取当前任务材料：

- `prd.md`
- 如果存在 `design.md`
- 如果存在 `implement.md`

```bash
python ./.trellis/scripts/get_context.py --mode packages
```

对每个发生改动的包/层，读取规范索引并遵循其中的**质量检查**部分：

```bash
cat .trellis/spec/<package>/<layer>/index.md
```

读取索引引用的具体规范文件；索引只是指针，不是终点。

## 步骤 3：运行项目检查

运行项目的 lint、类型检查和测试命令。修复所有失败后再继续。

## 步骤 4：按清单审查

### 代码质量

- [ ] Linter 是否通过？
- [ ] 类型检查是否通过（如适用）？
- [ ] 测试是否通过？
- [ ] 是否没有遗留调试日志？
- [ ] 是否没有压制警告或绕过类型安全？

### 测试覆盖

- [ ] 新函数 → 是否添加单元测试？
- [ ] 缺陷修复 → 是否添加回归测试？
- [ ] 行为变更 → 是否更新现有测试？

### 规范同步

- [ ] `.trellis/spec/` 是否需要更新（新模式、约定、经验教训）？

> “如果我修复了缺陷或发现了不直观的事项，是否应该记录下来，避免以后再次遇到？”→ 如果是，更新相关规范文档。

## 步骤 5：跨层维度（如适用）

如果改动仅限单层，跳过此步骤。

### A. 数据流（改动涉及 3 层或更多）

- [ ] 读取流是否正确贯通：存储 → 服务 → API → UI
- [ ] 写入流是否正确贯通：UI → API → 服务 → 存储
- [ ] 类型/Schema 是否在各层间正确传递？
- [ ] 错误是否正确传播给调用方？

### B. 代码复用（修改常量、创建工具函数）

- [ ] 创建新代码前是否搜索过现有相似实现？
  ```bash
  grep -r "pattern" src/
  ```
- [ ] 如果两个或更多位置定义同一值 → 是否提取为共享常量？
- [ ] 批量修改后是否更新了所有出现位置？

### C. 导入/依赖（创建新文件）

- [ ] 导入路径是否正确（相对路径与绝对路径）？
- [ ] 是否没有循环依赖？

### D. 同层一致性

- [ ] 使用同一概念的其他位置是否保持一致？

---

## 步骤 6：报告并修复

报告发现的违规项并直接修复。修复后重新运行项目检查。
