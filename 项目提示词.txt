在当前工作目录中从零创建并完成一个名为 Codex Relay 的 Windows 桌面软件。项目目录名为 `codex-relay`。

请持续完成项目初始化、架构设计、编码、测试、修复、构建和文档编写，不要只创建项目脚手架，也不要在完成部分页面后停止。

最终目标是生成一个能够在 Windows 10 和 Windows 11 上运行的轻量桌面程序，并成功生成 Release 可执行文件和 NSIS 安装程序。

# 一、项目基本信息

产品名称：

```text
Codex Relay
```

项目目录：

```text
codex-relay
```

可执行文件名称：

```text
CodexRelay.exe
```

软件定位：

这是一个用于管理 Windows 用户本机 Codex 配置的轻量工具，主要管理：

```text
~/.codex/config.toml
~/.codex/auth.json
```

软件特点：

1. 操作简单；
2. 启动速度快；
3. 常驻 Windows 系统托盘；
4. 支持开机自动启动；
5. 支持读取和管理已有的 `model_providers`；
6. 支持新增、编辑和删除 `model_providers`；
7. 支持快速切换当前使用的 Provider；
8. 切换时自动修改 `config.toml` 和 `auth.json`；
9. 启动时执行必要的自检；
10. 所有修改都要提示成功或失败；
11. 切换成功后必须提示用户重启 Codex 后生效。

# 三、Codex 配置目录

软件需要自动确定 Codex 配置目录。

解析顺序：

1. 如果设置了环境变量 `CODEX_HOME`，使用该目录；
2. 否则使用：

```text
%USERPROFILE%\.codex
```

默认文件路径：

```text
%USERPROFILE%\.codex\config.toml
%USERPROFILE%\.codex\auth.json
```

不要在代码中写死用户名。

开发和自动化测试时，允许使用以下环境变量覆盖 Codex 配置目录：

```text
CODEX_RELAY_CODEX_HOME
```

解析优先级：

```text
CODEX_RELAY_CODEX_HOME
CODEX_HOME
%USERPROFILE%\.codex
```

其中：

* `CODEX_RELAY_CODEX_HOME` 只用于开发和测试；
* 正式程序默认不要求用户设置该变量；
* 所有自动化测试必须使用临时目录；
* 任何测试都不得读取或修改真实的 `%USERPROFILE%\.codex`。

# 四、应用数据目录

软件自己的数据存放在：

```text
%LOCALAPPDATA%\CodexRelay\
```

目录结构：

```text
%LOCALAPPDATA%\CodexRelay\
├─ providers.json
├─ settings.json
├─ backups\
└─ logs\
```

各文件用途：

## 4.1 providers.json

保存所有 Provider 对应的 API Key。

## 4.2 settings.json

只保存软件设置，例如：

* 是否开机启动；
* 启动时是否显示主窗口；
* 是否关闭窗口时隐藏到托盘；
* 窗口大小；
* 窗口位置；
* 是否完成首次启动引导。

## 4.3 backups

保存配置修改前的备份。

## 4.4 logs

保存经过脱敏处理的软件运行日志。

# 五、配置文件职责

## 5.1 config.toml

`config.toml` 是 Provider 配置的主要数据来源。

已有 Provider 的配置可能类似：

```toml
model_provider = "provider-a"

[model_providers.provider-a]
name = "Provider A"
base_url = "https://provider-a.example.com/v1"
wire_api = "responses"

[model_providers.provider-b]
name = "Provider B"
base_url = "https://provider-b.example.com/v1"
wire_api = "responses"
```

字段名称必须是：

```toml
wire_api
```

不是：

```toml
wrie_api
```

软件需要读取和管理：

* 顶层 `model_provider`；
* 所有 `[model_providers.<provider_id>]`；
* 每个 Provider 的 `name`；
* 每个 Provider 的 `base_url`；
* 每个 Provider 的 `wire_api`；
* 可选的默认模型配置；
* 其他已有未知字段必须保留。

## 5.2 auth.json

当前使用 Provider 的 API Key 写入：

```json
{
  "OPENAI_API_KEY": "当前 Provider 的 API Key"
}
```

切换 Provider 时，需要自动将目标 Provider 的 API Key 写入 `auth.json`。

## 5.3 providers.json

所有 Provider 的 API Key 保存在：

```text
%LOCALAPPDATA%\CodexRelay\providers.json
```

格式：

```json
{
  "version": 1,
  "providers": {
    "provider-a": {
      "apiKey": "key-for-provider-a"
    },
    "provider-b": {
      "apiKey": "key-for-provider-b"
    }
  }
}
```

规则：

1. `providers.json` 只保存 Provider ID 和 API Key；
2. 不在 `providers.json` 中重复保存 `name`、`base_url` 和 `wire_api`；
3. Provider 配置以 `config.toml` 为准；
4. Provider 密钥以 `providers.json` 为准；
5. 当前活动密钥以 `auth.json` 为准；
6. API Key 使用明文保存，这是本项目明确接受的设计；
7. 不使用 Windows Credential Manager；
8. 不使用 Keyring；
9. 不使用 DPAPI；
10. 不使用 Stronghold；
11. 不需要实现复杂的密钥迁移或加密系统；
12. API Key 不得出现在日志、错误提示和 Git 仓库中。

如果 `providers.json` 不存在，启动时创建：

```json
{
  "version": 1,
  "providers": {}
}
```

如果 `providers.json` 已损坏：

1. 不要直接覆盖；
2. 先备份损坏文件；
3. 在界面显示错误；
4. 引导用户重新设置 API Key。

# 六、Provider 数据结构

前端至少使用以下数据结构：

```typescript
interface ProviderProfile {
  id: string;
  name: string;
  baseUrl: string;
  wireApi: "responses";
  model?: string;
  apiKeyConfigured: boolean;
  isActive: boolean;
  isValid: boolean;
  validationMessage?: string;
}
```

字段说明：

* `id`：对应 `[model_providers.<id>]`；
* `name`：对应 `name`；
* `baseUrl`：对应 `base_url`；
* `wireApi`：对应 `wire_api`；
* `model`：可选的默认模型；
* `apiKeyConfigured`：`providers.json` 中是否保存了非空密钥；
* `isActive`：顶层 `model_provider` 是否等于该 Provider ID；
* `isValid`：Provider 配置是否合法；
* `validationMessage`：配置异常时的说明。

普通 Provider 列表接口不得返回全部明文 API Key。

API Key 只能通过专门的编辑接口读取或修改。

API Key 输入框默认使用密码输入模式，并提供普通的显示或隐藏按钮。

# 七、启动流程

软件启动时必须尽快创建托盘图标。

启动流程分为关键自检和后台扩展自检。

## 7.1 启动关键自检

以下检查在启动阶段执行：

1. 确认只运行一个实例；
2. 解析 Codex 配置目录；
3. 检查配置目录是否存在；
4. 如果目录不存在，尝试创建；
5. 检查配置目录是否可读；
6. 检查配置目录是否可写；
7. 检查 `config.toml` 是否存在；
8. 检查 `auth.json` 是否存在；
9. 检查应用数据目录是否存在；
10. 检查 `providers.json` 是否存在；
11. 检查 `settings.json` 是否存在；
12. 检查备份目录是否存在；
13. 检查日志目录是否存在；
14. 尽快创建系统托盘；
15. 加载软件设置；
16. 加载 Provider 列表；
17. 判断当前 Provider。

启动阶段禁止执行网络请求。

启动阶段禁止自动访问 Provider 的 Base URL。

启动阶段禁止自动调用模型接口。

## 7.2 后台扩展自检

托盘创建后，在后台执行：

1. 解析 `config.toml`；
2. 解析 `auth.json`；
3. 解析 `providers.json`；
4. 检查 TOML 是否有效；
5. 检查 JSON 是否有效；
6. 检查顶层 `model_provider` 是否存在；
7. 检查当前 `model_provider` 是否存在于 Provider 列表；
8. 检查每个 Provider 是否包含 `name`；
9. 检查每个 Provider 是否包含 `base_url`；
10. 检查每个 Provider 的 `base_url` 是否是合法的 HTTP 或 HTTPS URL；
11. 检查每个 Provider 是否包含 `wire_api`；
12. 检查 `wire_api` 是否等于 `responses`；
13. 检查当前 Provider 是否存在 API Key；
14. 检查 `auth.json` 是否存在 `OPENAI_API_KEY`；
15. 检查当前 `auth.json` 中的密钥是否与 `providers.json` 中当前 Provider 的密钥一致；
16. 检查 `codex` 命令是否存在；
17. 在合理超时内执行 `codex --version`；
18. 检查开机启动实际状态；
19. 检查开机启动状态与软件设置是否一致；
20. 检查是否存在失败或未完成的写入事务；
21. 检查备份数量；
22. 检查配置文件是否被外部修改。

自检结果等级：

```text
正常
警告
错误
```

主界面显示：

```text
Codex 配置：正常
当前 Provider：Provider A
API Key：已配置
Codex CLI：已检测
开机启动：已启用
```

Codex CLI 未安装只属于警告，不能阻止配置管理功能使用。

# 八、首次启动处理

如果第一次启动时：

* `config.toml` 不存在；
* 或没有任何 `model_providers`；

软件应显示首次引导页面。

首次引导提供：

1. 打开 Codex 配置目录；
2. 新增第一个 Provider；
3. 退出；
4. 稍后设置。

软件不能自动创建带有虚假地址的 Provider。

可以创建空的 `providers.json` 和 `settings.json`，但不要自动创建虚假的 `config.toml` Provider。

# 九、读取已有 Provider

启动后读取：

```toml
[model_providers.*]
```

对每个 Provider 展示：

* Provider 名称；
* Provider ID；
* Base URL；
* Wire API；
* 默认模型；
* API Key 是否已设置；
* 是否为当前 Provider；
* 配置是否有效。

如果 `config.toml` 中已经存在 Provider，但 `providers.json` 中没有对应 API Key：

```text
API Key：未设置
```

如果当前 `auth.json` 中已经有：

```json
{
  "OPENAI_API_KEY": "xxx"
}
```

并且当前 Provider 在 `providers.json` 中没有密钥，则首次启动时可以提示：

```text
检测到当前 Codex 配置中存在 API Key，是否将其保存到当前 Provider？
```

只有用户确认后，才将该值写入 `providers.json`。

不要把该密钥自动关联到所有 Provider。

# 十、新增 Provider

提供“新增 Provider”功能。

输入字段：

1. Provider ID；
2. 显示名称；
3. Base URL；
4. Wire API；
5. 默认模型，可选；
6. API Key；
7. 保存后立即使用，可选。

校验规则：

## 10.1 Provider ID

* 必填；
* 只能包含英文字母、数字、下划线和连字符；
* 不允许空格；
* 不允许重复；
* 建议转换为小写；
* 不允许使用明显危险字符；
* 不允许包含 TOML 路径注入字符；
* 第一版创建后不可修改。

## 10.2 Name

* 必填；
* 去除首尾空格；
* 长度合理；
* 不能只包含空格。

## 10.3 Base URL

* 必填；
* 必须是有效的 HTTP 或 HTTPS URL；
* 去除首尾空格；
* 不自动猜测或随意添加路径；
* 保存用户输入的规范化结果。

## 10.4 Wire API

当前只允许：

```text
responses
```

界面可以直接固定显示：

```text
responses
```

不需要让用户输入其他值。

## 10.5 默认模型

* 可选；
* 不为空时去除首尾空格；
* 不限制必须是某种固定命名格式。

## 10.6 API Key

* 必填；
* 去除意外的首尾换行；
* 不要求必须以 `sk-` 开头；
* 不在日志中记录；
* 不在错误信息中显示；
* 不保存到前端 localStorage；
* 不保存到 Git 仓库。

保存后，在 `config.toml` 中新增：

```toml
[model_providers.<provider-id>]
name = "<Provider Name>"
base_url = "<Base URL>"
wire_api = "responses"
```

如果应用实际需要为 Codex API Key 文件认证增加其他必要字段，应集中在配置服务中实现，并在 README 中说明，不要在不同页面重复拼接配置。

API Key 保存到：

```json
{
  "version": 1,
  "providers": {
    "<provider-id>": {
      "apiKey": "<API Key>"
    }
  }
}
```

新增时必须保留：

* 其他 Provider；
* 其他 API Key；
* `config.toml` 中的所有无关配置；
* `config.toml` 中的注释；
* 未识别字段；
* MCP 配置；
* features 配置；
* sandbox 配置；
* profiles 配置；
* 用户其他自定义内容。

新增成功提示：

```text
Provider「{Provider Name}」已保存。
```

如果选择“保存后立即使用”，保存成功后继续执行切换流程。

成功提示：

```text
Provider「{Provider Name}」已保存并启用。请重启 Codex 后生效。
```

# 十一、编辑 Provider

允许编辑：

* Name；
* Base URL；
* 默认模型；
* API Key。

第一版不允许直接修改 Provider ID。

界面提示：

```text
Provider ID 创建后不可修改。
```

编辑流程：

1. 重新读取最新 `config.toml`；
2. 重新读取最新 `providers.json`；
3. 检查文件是否被外部修改；
4. 更新目标 Provider 的字段；
5. 更新目标 Provider 的 API Key；
6. 保留其他 Provider；
7. 保留其他 API Key；
8. 保留 TOML 注释；
9. 保留 TOML 未知字段；
10. 使用临时文件写入；
11. 校验写入结果；
12. 替换正式文件；
13. 失败时恢复原文件；
14. 刷新界面和托盘菜单。

API Key 编辑规则：

* 输入框默认掩码；
* 提供显示或隐藏按钮；
* 用户没有修改 API Key 时，不要覆盖原密钥；
* 用户明确清空 API Key 时，需要二次确认；
* 当前正在使用的 Provider 如果被清空 API Key，显示警告。

编辑成功提示：

```text
Provider「{Provider Name}」已更新。
```

如果编辑的是当前 Provider：

```text
Provider「{Provider Name}」已更新。请重启 Codex 后生效。
```

编辑当前 Provider 时，如果 Base URL、Name、Wire API、默认模型或 API Key 发生变化，应询问：

```text
是否立即将最新配置写入 Codex？
```

用户确认后，执行完整切换事务。

# 十二、删除 Provider

支持删除非当前 Provider。

删除前显示确认：

```text
确定删除 Provider「{Provider Name}」吗？

该操作将删除：
1. config.toml 中对应的 Provider 配置；
2. providers.json 中对应的 API Key。

其他 Provider 不受影响。
```

规则：

1. 当前正在使用的 Provider 不允许直接删除；
2. 当前 Provider 删除按钮禁用；
3. 提示用户先切换到其他 Provider；
4. 删除前创建备份；
5. 从 `config.toml` 删除对应 `[model_providers.<id>]`；
6. 从 `providers.json` 删除对应密钥；
7. 保留其他配置；
8. 保留其他密钥；
9. 删除失败时恢复；
10. 删除成功后刷新托盘菜单。

删除成功：

```text
Provider「{Provider Name}」已删除。
```

删除失败：

```text
删除失败：{安全且可理解的错误原因}。原配置已恢复。
```

# 十三、切换 Provider

切换 Provider 是软件最重要的功能。

用户可以通过：

* 主窗口；
* 系统托盘菜单；

选择目标 Provider。

主窗口和托盘菜单必须调用同一个 Rust 后端切换服务。

前端不得自行拼接 TOML 或直接写文件。

完整切换流程如下。

## 13.1 切换前检查

1. 获取全局配置写入互斥锁；
2. 防止两个切换操作同时执行；
3. 重新读取磁盘上的 `config.toml`；
4. 重新读取磁盘上的 `auth.json`；
5. 重新读取磁盘上的 `providers.json`；
6. 校验目标 Provider 是否存在；
7. 校验目标 Provider 的 Name；
8. 校验目标 Provider 的 Base URL；
9. 校验目标 Provider 的 Wire API；
10. 检查目标 Provider 是否存在 API Key；
11. 如果 API Key 缺失或为空，停止切换；
12. 检查三个文件是否被其他程序修改；
13. 创建同一事务编号的备份。

缺少密钥时提示：

```text
该 Provider 尚未设置 API Key，无法启用。
```

## 13.2 修改 config.toml

切换时修改顶层：

```toml
model_provider = "<目标 Provider ID>"
```

确保目标 Provider 配置包含：

```toml
[model_providers.<目标 Provider ID>]
name = "<Provider Name>"
base_url = "<Base URL>"
wire_api = "responses"
```

如果目标 Provider 配置了默认模型，则同步修改顶层：

```toml
model = "<目标 Provider 的默认模型>"
```

如果目标 Provider 没有设置默认模型：

* 不修改原有顶层 `model`；
* 不删除原有顶层 `model`。

如果 Codex 使用 `auth.json` API Key 模式需要：

```toml
cli_auth_credentials_store = "file"
```

则切换时确保该字段正确写入。

不要改动：

* 其他 Provider；
* MCP 配置；
* features；
* sandbox；
* profiles；
* history；
* 用户自定义字段；
* 未识别字段；
* 注释和无关空行。

## 13.3 修改 auth.json

从 `providers.json` 读取目标 Provider 的 API Key。

将 `auth.json` 写成：

```json
{
  "OPENAI_API_KEY": "<目标 Provider 的 API Key>"
}
```

要求：

* UTF-8；
* 两个空格缩进；
* 文件末尾有换行；
* 不在日志中输出密钥；
* 不在通知中输出密钥；
* 不在错误中输出完整文件内容。

## 13.4 安全写入

不能直接打开正式文件后清空并写入。

使用以下流程：

1. 创建备份；
2. 读取最新文件；
3. 在内存中生成修改结果；
4. 写入同目录临时文件；
5. flush；
6. 必要时同步文件；
7. 重新解析临时 TOML；
8. 重新解析临时 JSON；
9. 验证内容；
10. 使用可靠的替换方式替换正式文件；
11. 重新读取正式文件；
12. 再次验证。

验证项目：

* 顶层 `model_provider` 等于目标 Provider ID；
* 目标 Provider 配置存在；
* `name` 正确；
* `base_url` 正确；
* `wire_api` 等于 `responses`；
* 如果存在默认模型，顶层 `model` 正确；
* `auth.json.OPENAI_API_KEY` 存在；
* `auth.json.OPENAI_API_KEY` 与 `providers.json` 中目标密钥一致；
* TOML 可以正常解析；
* JSON 可以正常解析。

验证日志不得输出 API Key。

## 13.5 失败回滚

如果出现以下任一错误：

* `config.toml` 写入失败；
* `auth.json` 写入失败；
* 临时文件解析失败；
* 正式文件替换失败；
* 写入后验证失败；

必须恢复修改前的：

```text
config.toml
auth.json
```

如果 `providers.json` 在该操作中也被修改，则一并恢复。

失败提示：

```text
切换失败：{安全且可理解的错误原因}。原配置已恢复，未对 Codex 配置造成更改。
```

不得在无法确认回滚成功时错误提示“已恢复”。

如果回滚也失败，提示：

```text
切换失败，并且自动恢复未完全成功。请立即从备份页面恢复上一份配置。
```

## 13.6 成功提示

切换成功后：

1. 更新主界面；
2. 更新托盘菜单；
3. 更新当前 Provider 标记；
4. 记录不包含密钥的日志；
5. 显示 Windows 通知；
6. 显示应用内通知。

通知内容必须是：

```text

```

# 十四、配置写入事务

实现统一事务服务。

建议事务结构：

```rust
struct ConfigTransaction {
    id: String,
    operation: String,
    provider_id: Option<String>,
    started_at: String,
}
```

可能的操作类型：

```text
create_provider
update_provider
delete_provider
switch_provider
restore_backup
sync_current_provider
```

所有涉及多个文件的操作必须集中在事务服务中。

不能在不同 Tauri Command 中分别随意写入文件。

使用全局异步互斥锁或其他可靠机制，防止并发写配置。

# 十五、备份与恢复

备份目录：

```text
%LOCALAPPDATA%\CodexRelay\backups\
```

备份结构：

```text
backups/
└─ 20260720-220000-<transaction-id>/
   ├─ config.toml
   ├─ auth.json
   ├─ providers.json
   └─ metadata.json
```

如果原文件不存在，应在 `metadata.json` 中记录，而不是创建误导性的空文件。

`metadata.json` 可以保存：

```json

  "transactionId": "xxx",
  "createdAt": "2026-07-20T22:00:00+08:00",
  "operation": "switch_provider",
  "providerId": "provider-a",
  "configExisted": true,
  "authExisted": true,
  "providersExisted": true,
```

不得在 `metadata.json` 中保存 API Key。

备份要求：

1. 每次修改前自动备份；
2. 最多保留最近 20 份事务备份；
3. 超出后删除最旧备份；
4. 不能删除当前正在使用的事务备份；
5. 支持手动恢复；
6. 恢复前再次备份当前状态；
7. 恢复后重新执行自检；
8. 恢复后刷新 Provider 列表；
9. 恢复后刷新托盘菜单；
10. 不在界面直接展示备份文件中的完整 API Key。

注意：

* `auth.json` 和 `providers.json` 中包含明文 API Key；
* 这是本项目接受的简化设计；
* README 中必须明确说明这一点。

# 三十四、遇到问题时的处理方式

不要因为以下非关键问题停止整个 Goal：

* UI 间距不确定；
* 图标暂时没有最终版本；
* 文案有多种合理表达；
