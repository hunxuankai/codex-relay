# 设计：统一内部文档与提交信息中文规范

## 变更边界

本任务分为三类变更：

1. 文档翻译：处理 `.agents/**/*.md` 与 `.trellis/**/*.md` 的解释性自然语言。
2. AI/配置说明翻译：处理 `.codex/agents/*.toml` 的描述与提示词，以及 `.codex/config.toml`、`.trellis/config.yaml` 中的说明性注释。
3. 长期规则与自动提交：更新 `AGENTS.md`、工作流文档，以及 Trellis 自动提交主题的配置默认值和归档实现。

## 翻译契约

- Markdown 标题、正文、表头、清单说明、引用说明和面向人的示例文字使用简体中文。
- fenced code block、行内代码、命令、路径、文件名、配置键、占位符和协议字面量原样保留。
- Skill frontmatter 的键和 `name` 保持不变；`description` 可翻译为中文。
- Markdown 链接显示文字可翻译，链接目标不得改变；若文档存在显式标题锚点引用，翻译后同步校验。
- TOML/YAML 的键、枚举值和结构不变，只翻译注释、描述和多行提示词中的自然语言。
- 历史任务文档只翻译残留英文说明，不改写历史事实、命令输出、提交哈希或当时结论。

## Git 提交信息契约

- 人工提交采用 `<type>(<scope>): <中文主题>`，其中 `type` 与 `scope` 作为工具约定可保留英文。
- 提交正文使用简体中文；命令、路径、标识符和必要技术名词可保留英文。
- 会话日志自动提交主题由 `.trellis/config.yaml` 配置为中文，并同步中文默认值。
- 任务归档自动提交主题在 `task_store.py` 中改为 `chore(task): 归档 <task-name>`。

## 兼容性与风险

- 大批量翻译最主要的风险是误改代码块、配置结构、占位符或链接目标，因此按文件类型分批修改并在每批后做结构校验。
- `.agents` Skill 文档属于可执行工作流指令，翻译必须保持规范性词义与 MUST/禁止类约束强度。
- `.trellis/workflow.md` 是工作流核心，需保留条件块、HTML 注释、命令和阶段编号。
- 自动提交主题改变只影响新提交，不重写 Git 历史。

## 回滚策略

- 文档翻译按逻辑目录分批，可通过 Git diff 精确审查和逐文件回退。
- 自动提交主题变更独立成小范围代码与配置修改；若测试失败，先回退实现并保留文档规则，直到实现修复。
- 不执行会影响真实用户 Codex 配置或 `%LOCALAPPDATA%\\CodexRelay` 的操作。
