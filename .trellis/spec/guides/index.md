# 思考指南

> **用途**：扩展思考范围，发现可能遗漏的事项。

---

## 为什么需要思考指南？

**多数缺陷和技术债源于“没有想到这一点”**，而不是能力不足：

- 没有考虑层边界会发生什么 -> 跨层缺陷
- 没有考虑代码模式重复 -> 到处都是重复代码
- 没有考虑边界情况 -> 运行时错误
- 没有考虑未来维护者 -> 代码难以阅读

这些指南帮助你**在编码前提出正确的问题**。

## 速查：思考触发条件

### 何时考虑跨层问题

- [ ] 功能涉及 3 层或更多层（API、Service、Component、Database）
- [ ] 数据格式在层之间变化
- [ ] 多个消费者需要相同数据
- [ ] 不确定某段逻辑应放在哪里
- [ ] 正在添加事件 kind、JSONL 记录、RPC 载荷或配置字段
- [ ] UI / 命令代码开始直接转换原始载荷字段

-> 阅读[跨层思考指南](./cross-layer-thinking-guide.md)

### 何时考虑代码复用

- [ ] 正在编写与现有代码相似的代码
- [ ] 看到同一模式重复 3 次或更多
- [ ] 正在向多个位置添加新字段
- [ ] **正在修改任何常量或配置**
- [ ] **正在创建新的工具/辅助函数** <- 先搜索！
- [ ] 两个文件通过本地类型转换读取同一个无类型载荷字段
- [ ] 多个分支根据 `kind` / `action` 更新同一个派生状态

-> 阅读[代码复用思考指南](./code-reuse-thinking-guide.md)

### 验证 AI 交叉审查结果时

- [ ] 审查者声称“用户输入可能恶意” -> 检查真实数据来源（内部清单、用户配置还是外部 API？）
- [ ] 审查者指出“缺少验证” -> 数据是否来自可信内部来源？
- [ ] 审查者声称“行为变化” -> 阅读代码注释，是否属于有意设计？
- [ ] 审查者在测试中发现“缺陷” -> 在思维中删除被测功能，测试是否仍通过？如果是，就是同义反复测试

**AI 审查者常见误报模式**：
1. **混淆信任边界**：把内部数据（内置 JSON 清单）当作不可信外部输入
2. **忽略设计注释**：把代码注释中记录的有意行为标记为缺陷
3. **误读变量**：没有追踪变量的实际定义，例如混淆以路径或名称为键的 Map

**验证规则**：每个 CRITICAL/WARNING 发现都必须先对照真实代码验证，再确定优先级。对 AI 审查预留约 35% 的误报比例。

## 修改前规则（关键）

> **修改任何值前，始终先搜索！**

```bash
# Search for the value you're about to change
grep -r "value_to_change" .
```

这一习惯可以防止大多数“忘记更新 X”的缺陷。
